6 项
Handoff reply 实际发消息给用户
POST /api/v1/handoff/{task_id}/reply 目前是存根，仅返回 {"status":"sent"}。 需要把运营回复内容写入 messages 表，并通过 Telegram Bot API（或对应渠道）实际发送给用户。
Backend Telegram Bot
危机干预完整流程
检测到高危表达（self-harm / suicide / I want to die 等关键词）时：AI 立即切换安全话术 → 禁止商业引导 → 创建 P0 handoff 任务 → 写入 risk_events（risk_type='crisis'）→ 用户标记进入 S5。当前这整条流程代码不存在。
Backend 合规
风险事件实际写库（risk_events 表）
POST /api/v1/users/{id}/risk-events 当前只返回 {"status":"ok"}， 完全没有写入 risk_events 表的逻辑。 GET /api/v1/users/{id}/risk-events 查询接口也未实现。
Backend DB
Telegram 通知实际发送 Worker
silent_reactivation_scheduler 已将任务写入 notification_tasks 表， 但没有任何 Worker 消费这张表并调用 Telegram Bot API 发送消息。 用户永远收不到沉默回流通知。
Worker Backend
内容安全双层过滤（OpenAI Moderation API）
架构要求所有 AI 回复和用户输入经过：① OpenAI Moderation API（异步）② 关键词库（同步阻塞）。 当前 messages.safety_result 字段始终为 null，没有任何过滤执行。
Backend 合规
账号删除接口（GDPR 遗忘权）
POST /api/v1/users/{id}/delete-request 路由完全未注册。 需实现：接收删除申请 → 级联删除用户数据（或匿名化）→ 保留法规要求的最小审计记录 → 确认通知。

P1 — 重要（核心业务逻辑缺失）

6 项
接管自动触发 Policy Service
架构方案 §2.3 定义了 7 个触发条件（loneliness_score 超阈值、VIP 阈值更低、危机信号、 AI 置信度低、用户明确要求真人等），满足任一条件时自动创建 handoff_task 入队。 当前这个"Policy Service"判断逻辑不存在，接管完全依赖手动触发。
Backend Worker
关系阶段自动升降级逻辑（S0→S5）
user_profiles.relationship_stage 字段存在，但没有任何代码根据行为信号 （会话频率、平均长度、主动发起占比、付费行为等）自动升级或降级。 当前所有用户永远停留在 S0。
Backend Worker
S5 进入/退出限制实际执行
当 risk_score ≥ 70 进入 S5 后，应冻结所有商业推送、禁止 Upsell、 每 48-72 小时仅允许安全关怀、强制 P0 接管。 连续 7 天 risk_score < 40 后恢复。这些限制逻辑均未实现，高风险用户仍可收到营销推送。
Backend 合规
GDPR 字段对齐 + 欧盟强制 opt-in 流程
当前数据库用 notification_opt_in / opt_out_marketing， 而架构方案要求 marketing_opt_in / transactional_notifications_allowed / is_eu_user / region。字段名和语义不一致。 is_eu_user 字段在 init.sql 里不存在，EU 强制 opt-in 流程未实现。
DB Migration 合规 Backend
Handoff 任务分布式锁（5 分钟超时自动释放）
架构 §8.3 要求：每个任务只能被一个运营锁定（Redis 分布式锁）， 超过 5 分钟未处理自动释放重新入队，P0 任务可抢占低优先级任务。 当前 lock 接口只是一个简单 UPDATE，无分布式锁、无超时机制。
Backend Redis
risk_score → users.risk_level 派生更新
架构 §14.2 要求：写入 user_profiles.risk_score 后， 同事务更新 users.risk_level 分桶（0-39=normal, 40-69=elevated, 70-89=high, 90-100=crisis）。 当前写 risk_score 后 risk_level 永远不变。
Backend DB
